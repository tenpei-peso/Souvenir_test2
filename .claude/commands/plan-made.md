# Plan コマンド

このコマンドは、コードを書き始める前に **planner (プランナー)** を主担当として包括的な実装計画を作成し、必要に応じて **architect** の観点を併用します。

## このコマンドの機能

1. **要件の再確認 (Restate Requirements)** - 何を構築する必要があるかを明確にします。
2. **リスクの特定 (Identify Risks)** - 潜在的な問題やブロッカー（障害）を洗い出します。
3. **ステップ計画の作成 (Create Step Plan)** - 実装を複数のフェーズに分割します。
4. **確認の待機 (Wait for Confirmation)** - 作業を進める前に、必ずユーザーからの承認を待ちます。

## いつ使うべきか

以下のような場面で `/plan` を使用してください：

- 新しい機能の開発を始める時
- 大規模なアーキテクチャの変更を行う時
- 複雑なリファクタリングに取り組む時
- 複数のファイルやコンポーネントに変更が及ぶ時
- 要件が不明確、または曖昧な時

## 仕組み

プランナーエージェントは以下の手順を実行します：

1. リクエストを**分析**し、明確な言葉で要件を再確認します。
2. 具体的で実行可能なステップを含む**フェーズに分割**します。
3. コンポーネント間の**依存関係を特定**します。
4. **リスク**と潜在的なブロッカーを**評価**します。
5. **複雑さを予測**します（高/中/低）。
6. **計画を提示**し、ユーザーからの明確な承認を**待ちます**。

## 使用例

ユーザー: /plan マーケット（市場）が解決・確定した時に、リアルタイムで通知を送る機能を追加したいです。

# 実装計画: マーケット解決時のリアルタイム通知

## 要件の再確認

- ユーザーがウォッチしているマーケットが解決した際に通知を送信する
- 複数の通知チャネル（アプリ内、メール、Webhook）をサポートする
- 通知が確実に配信されるようにする
- マーケットの結果と、ユーザーのポジションの結果を含める

## 実装フェーズ

### フェーズ 1: データベーススキーマ

- notifications テーブルを追加（カラム: id, user_id, market_id, type, status, created_at）
- チャネル設定用の user_notification_preferences テーブルを追加
- パフォーマンス向上のため、user_id と market_id にインデックスを作成

### フェーズ 2: 通知サービス

- lib/notifications.ts に通知サービスを作成
- BullMQ/Redis を使用して通知キューを実装
- 配信失敗時の再試行（リトライ）ロジックを追加
- 通知テンプレートを作成

### フェーズ 3: 統合ポイント

- マーケットの解決ロジックにフックする（ステータスが "resolved" に変わった時）
- そのマーケットにポジションを持っている全ユーザーをクエリで取得
- 各ユーザーの通知をキューに追加

### フェーズ 4: フロントエンドコンポーネント

- ヘッダーに NotificationBell コンポーネントを作成
- NotificationList モーダルを追加
- Supabase のサブスクリプションを利用してリアルタイム更新を実装
- 通知設定ページを追加

## 依存関係

- Redis (キュー用)
- メールサービス (SendGrid/Resend)
- Supabase リアルタイムサブスクリプション

## リスク

- 高: メールの到達率 (SPF/DKIM の設定が必要)
- 中: 1マーケットあたり1000人以上のユーザーがいる場合のパフォーマンス
- 中: マーケットが頻繁に解決される場合の通知スパム
- 低: リアルタイムサブスクリプションのオーバーヘッド

## 予想される複雑さ: 中 (MEDIUM)

- バックエンド: 4-6 時間
- フロントエンド: 3-4 時間
- テスト: 2-3 時間
- 合計: 9-13 時間
