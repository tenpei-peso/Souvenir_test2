# コードレビュー

未コミットの変更を **code-reviewer** を主担当としてレビューし、必要に応じて **security-reviewer** の観点も合わせて確認します：

1. 変更されたファイルを取得：`git diff --name-only HEAD` と `git ls-files --others --exclude-standard` を組み合わせて tracked / untracked の両方を列挙

2. 変更された各ファイルについて、チェック：

**セキュリティ問題（重大）：**

- ハードコードされた認証情報、API キー、トークン
- SQL インジェクション脆弱性
- XSS 脆弱性
- 入力検証の不足
- 不安全な依存関係
- パストラバーサルリスク

**コード品質（高）：**

- 関数の長さが 50 行以上
- ファイルの長さが 800 行以上
- ネストの深さが 4 層以上
- エラーハンドリングの不足
- デバッグ用出力の残留
- `TODO`/`FIXME` コメント
- 公開 API にドキュメントがない

**ベストプラクティス（中）：**

- 可変パターン（イミュータブルパターンを使用しますすべき）
- コード/コメント内の絵文字使用します
- 新しいコードのテスト不足
- アクセシビリティ問題（a11y）

3. 以下を含むレポートを生成：
   - 重大度：重大、高、中、低
   - ファイル位置と行番号
   - 問題の説明
   - 推奨される修正方法

4. 重大または高優先度の問題が見つかった場合、コミットをブロック

セキュリティ脆弱性を含むコードは絶対に許可しないこと！
